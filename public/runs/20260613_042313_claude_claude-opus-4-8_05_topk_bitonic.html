Key findings & design

The decisive insight from profiling: inputs are tiny (0.5–2 MB), so this is latency/launch-bound, not bandwidth-bound — the pure read floor is ~8µs at one wave of blocks (~188), and a trivial kernel can't exceed it. So the game is minimizing wall-time overhead.

The kernel is a hybrid dispatched by k:

  • k=1: dedicated block-argmax reduction, single launch (bpr=1).
  • k≤8: per-thread register top-k (threshold-gated, unrolled templated insertion) + pairwise tree-merge in shared memory.
  • k≥16: bitonic sort in shared memory with (value,index) packed into one uint64 key (≈1.5× speedup — one 64-bit compare-swap instead of two 32-bit), using a multi-pass reduction: tiny phase-1 chunks (256) → many blocks fill the SMs with less total log² work → grid of merge passes shrink candidates.
