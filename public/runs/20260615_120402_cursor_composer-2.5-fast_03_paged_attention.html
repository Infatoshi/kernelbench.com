Implemented a Triton paged-attention decode kernel in solution.py. It passes check.py and reaches ~24% of peak DRAM bandwidth (geomean peak_fraction: 0.2399, well above the 0.1 threshold).

Approach

The kernel uses a GQA-aware, page-batched design:

  • Grid: (batch, num_kv_heads) — one CTA per batch element and KV head, sharing KV loads across group_size query heads
  • Page loop: walks block_table to gather non-contiguous pages from the global KV pool
  • Per-page batching: loads a full page (page_size=16, head_dim) of K/V in one shot via 2D blocked loads
  • Matmul: tl.dot for QK (group_size × head_dim @ head_dim × page_size) and PV
  • Numerics: online softmax in fp32 (FlashAttention-style), bf16 tensor-core dots, fp32 accumulation — matches the reference within 0.02 tolerance

Key layout handling

  • KV cache last dim packs [K | V] — K at [:head_dim], V at [head_dim:]
  • Partial final pages are masked with tokens_in_page (covers non-power-of-2 seq lengths like 1535 and 2000)
  • GQA broadcast is implicit: heads kv_head * group_size .. +group_size share the same K/V stream

Performance (representative)

| Shape | seq_len | Peak fraction | |-------|---------|---------------| | 0 (8B decode) | 1024 | 21% | | 1 (batched) | 2048 | 73% | | 2 (70B long) | 4096 | 15% | | 3 (non-pow2) | 1535 | 44% | | 4 (small D) | 2000 | 8% |

Shape 2 is loop-bound (256 page iterations); shape 4 has lower occupancy with head_dim=64. Both still contribute to a passing geomean.
