Key design decisions (all empirically validated on SM120)

  1. TMA ragged descriptors (triton.tools.ragged_tma) for the variable-length grouped M dimension — hardware bounds-checking, zero mask work in the K-loop, and cp.async.bulk loads.
  2. BN=256 + num_warps=8 is what saturates Blackwell tensor cores (single GEMM hits 379 TFLOPS ≈ cuBLAS). 4 warps spills the 256-wide accumulator and collapses to ~5 TFLOPS.
  3. The bf16-gate trick to fuse both GEMMs into one kernel: a naive single pass needs two BN=256 fp32 accumulators → register spill. Instead, reduce the gate accumulator to bf16 (halving its footprint) before the up K-loop, so ≤1.5 accumulators are live. One launch, gate never touches HBM — beats both true-fusion (spills) and two-separate-GEMMs (extra launch + gate HBM round-trip).
  4. Grid (num_N, num_M, E); max_n_e cached by (data_ptr, T_perm, E) (data_ptr alone is unsafe — CUDA address reuse collided shapes sharing E+1 ints).

Key findings worth noting

  • Triton 3.6 lowers bf16 tl.dot to mma.sync, not tcgen05, on SM120 even with TMA. Despite that, it reaches ~96% of cuBLAS, so the big-effort tcgen05/CUTLASS path isn't worth it.
  • The real bf16 peak is ~400 TFLOPS (cuBLAS), not the 200 in hardware.py — the scoring metric divides by 200, so it's structurally capped near actual_util / K / peak.
  • The remaining gap vs cuBLAS on the big shapes is the cold-L2 x-read-twice penalty (benchmark flushes 128 MB L2). Only tcgen05 true single-pass fusion (x once, accumulators in tmem via Gluon) would close it — ~+0.02 for expert-level effort; I left it as documented future work in memory.
