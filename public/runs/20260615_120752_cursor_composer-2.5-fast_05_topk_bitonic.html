Implementation

Custom CUDA kernel (via torch.utils.cpp_extension.load_inline, SM120) with three paths:

  1. k=1 (argmax) — vectorized float4 loads, warp shuffle reduction, one block per row. ~193 GB/s on shape 4 (128×4096), beating torch.topk (~156 GB/s).
  1. k>1, batched rows — one block per row; each thread streams the row with float4 loads, maintains a sorted top-k in registers, then tree-merges in shared memory (power-of-2 thread counts: 128 for k≤16, 64 for k=32/64).
  1. Large vocab (1×131072×64) — multi-block slice pass (up to 1024 blocks) + single-block merge, reusing cached partial workspace.

Performance (benchmark)

| Shape | Config | GB/s | vs torch.topk | |-------|--------|------|---------------| | 0 | 1×131072×64 | ~0.28 | ~30× slower (compute-bound selection) | | 1 | 64×8192×8 | ~55 | Matches | | 2 | 32×16384×32 | ~2.4 | ~23× slower (O(n·k) inserts) | | 3 | 16×12000×16 | ~4.0 | Moderate | | 4 | 128×4096×1 | ~193 | Beats reference |

Geomean peak fraction ≈ 0.4% of DRAM roofline — dominated by shapes 0 and 2 where per-element top-k insertion is expensive compared to PyTorch’s radix-select internals.

What was tried

  • 8-bit radix selection — faster in theory, but subtle correctness bugs (histogram init, tie handling) prevented reliable PASS across all shapes/seeds.
  • Tiled bitonic sort for k≥32 — regressed on shape 2.

The shipping solution prioritizes correctness across all shapes, seeds, and numeric stress cases while matching or beating torch.topk on the batched small-k and argmax shapes. Closing the gap on the decoder-vocab (131K×64) and mid-k (32×16384×32) shapes would need a correct GPU radix-select or CUB-based partial sort — the main direction for further work.
