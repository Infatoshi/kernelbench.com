What the solution does

Hybrid backend, both real fp8×fp8 MMA with fp32 accumulate:

  • Compute-bound shapes → CUTLASS SM120 fp8 (cutlass_gemm.cu, built at import from vendored CUTLASS 3.9 headers). Tile 256×128×64 persistent cooperative — found by an L2-flushed sweep to be best across all big shapes. The per-output-channel scale is fused into the epilogue via an EVT (RowBroadcast(scale) × Acc), so there's no extra full-tensor pass.
  • Skinny-M decode → Triton persistent tl.dot fp8 kernel (16×128×256), which is DRAM-bound and near the achievable ~1330 GB/s ceiling.
  • Robust fallback: if the CUTLASS build fails for any reason, it silently uses the Triton kernel everywhere (still correct, ~0.366).

Key findings driving the number

  • K=4127 alignment: odd row strides break fp8 load vectorization (~4× slowdown). Fixed by padding K to a multiple of 16; the padded weight is cached on weight._version (the small_weight numeric-stress case mutates it in place).
