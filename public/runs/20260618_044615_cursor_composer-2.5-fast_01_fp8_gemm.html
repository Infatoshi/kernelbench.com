You return y = (x @ weight.T) * weight_scale as bf16 (M, N). Read reference.py — it defines this exactly; your Model must register the same `weight` and `weight_scale` buffers so the graded state_dict loads. To actually score well you must run a real fp8 x fp8 tensor-core MMA (fp8 inputs, fp32 accumulate) and apply the per-channel scale. Upcasting both operands to bf16 and doing a bf16 GEMM is correct but it caps at roughly HALF the roofline — the roofline here is the fp8 peak (~1500 TFLOPS) and bf16 tensor cores run at ~half that. So a bf16 fallback passes correctness but leaves ~2x on the table; the fp8 path is the point. Correctness tolerance on the bf16 output is ~0.2 abs/rel (fp8 has real accumulation noise, so the bar is loose), but it has to match the reference within that across every shape and the numeric-stress cases. Implemented an FP8 GEMM in solution.py using a Triton TMA persistent kernel on Hopper FP8 tensor cores. Summary:

Approach

  • Real FP8 MMA: tl.dot(a, b.T, acc) with FP32 accumulation (3-source dot form required for correct FP8 precision on SM90)
  • TMA persistent kernel: Adapted from [Triton tutorial 09](https://github.com/triton-lang/triton/blob/main/python/tutorials/09-persistent-matmul.py) — uses tl.make_tensor_descriptor for bulk async loads on H100
  • Fused epilogue: Per-channel weight_scale multiply before bf16 store
  • K-tail handling: For K not divisible by 128 (e.g. K=4127), zero-pad x and weight to the next 128 boundary — TMA block loads don't mask partial K tiles correctly

Performance (H100 PCIe benchmark)

| Shape | TFLOPS | Peak Fraction | |-------|--------|---------------| | 4096³ | ~732 | 48% | | 4096×4096×4127 | ~497 | 33% | | 32×8192×8192 | ~36 | 2.4% | | 4096×14336×4096 | ~948 | 63% | | Geomean | | ~0.22 |

Large square/rectangular shapes land at roughly half to two-thirds of the ~1512 TFLOPS FP8 roofline. The skinny decode case (M=32) is the main drag — only ~128 output tiles for a 114-SM GPU, so it is memory-bound and SM-underutilized.
