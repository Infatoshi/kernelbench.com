This is the monorepo for the kernelbench.com website + the benchmark suites it visualizes. If you're a coding agent picking this up cold, read this whole file before you touch anything.

For the why behind the design, read [`benchmarks/hard/DEVLOG.md`](./benchmarks/hard/DEVLOG.md). The most recent entry is the launch-prep summary and is the single best place to catch up.

## Non-negotiable rules

### Vercel deploy gotcha (THE MOST IMPORTANT ONE)

**Commits must be authored as `elliot@arledge.net`.** Vercel's commit-verification gate silently fails any commit with an unverified author email at the pre-build phase — no build logs, just a status of `failure`. The local `git config user.email` is already set in this repo, so plain `git commit` should work. If you're scripting commits with `-c user.email=...`, use `elliot@arledge.net`. The autogenerated tailnet hostname (`infatoshi@anvil.tail21a94e.ts.net`) WILL fail.

### Auto-deploy

`master` auto-deploys on push via Vercel's native GitHub integration. There's no GitHub Actions workflow. Project: `kernelbench` under team `elliot-arledges-projects`, project ID `prj_BFoWvlg7vgjIKGBaK7CTjOaIIenC`, team ID `team_0HxxgHRPOylQlRSJybcCgBAl`.

### The website reads from the filesystem

`lib/data.ts` does `readFile()` at build time. No HTTP fetch, no ISR. To update what the site shows: change the file under `benchmarks/`, push, Vercel rebuilds. Don't introduce `fetch()` to GitHub raw URLs as a workaround for not-yet-committed data — commit the data first.

### Per-suite editing

When working on `benchmarks/hard/` or `benchmarks/v3/`:
- Read [`benchmarks/hard/CLAUDE.md`](./benchmarks/hard/CLAUDE.md) for that suite's specific rules.
- **uv only.** No bare `python`, no `pip`. Use `uv run ...`, `uv add ...`, `uv pip install ...`.
- Before committing benchmark code: `cd benchmarks/hard && uv run ruff check . --fix && uv run pytest`.
- Never edit `problems/*/solution.py` (they're agent output, gitignored).
- Never modify `problems/*/{reference,check,benchmark}.py`, `problem.yaml`, `shapes.py`, `PROMPT.txt` after a sweep is published — those define the benchmark, changing them invalidates prior results.
- See `benchmarks/hard/DEVLOG.md` for the rubric-leak findings — they're the headline. Two of seven problems leak the rubric; we publish anyway with leaks documented inline.

## Common tasks

### Develop the site locally

```bash
npm install
npm run dev
# http://localhost:3000
```

### Regenerate the 100 transcript viewers

If `outputs/runs/` (gitignored, lives on the GPU machine at `~/cuda/KernelBench-Hard/outputs/runs/`) has new transcripts, regenerate the public viewers:

```bash
cd benchmarks/hard
RUN_SRC=/path/to/outputs/runs
ANNOT_DIR=$(pwd)/results/annotations
for d in "$RUN_SRC"/*/; do
  [ -f "$d/transcript.jsonl" ] || continue
  rid=$(basename "$d")
  KB_VIEWER_THEME=phosphor KB_ANNOTATIONS_DIR="$ANNOT_DIR" \
    uv run --quiet python -m src.viewer "$d" --out ../../public/runs/"$rid".html
done
```

### Regenerate baseline timings

```bash
cd benchmarks/hard
bash scripts/run_baselines.sh
cp results/problem_baselines.json ../../benchmarks/hard/results/problem_baselines.json
```

### Regenerate blog plots

```bash
cd benchmarks/hard
python3 scripts/generate_blog_plots.py
# writes to ../../public/blog-hard/*.png
```

### Add a new annotation

Just create `benchmarks/hard/results/annotations/<run_id>.yaml` matching the schema in `SCHEMA.md`. The website picks it up at next build; the `/hard` cell gets a `★` marker; the run viewer gets a "reward hack" tab if the annotation file exists.
