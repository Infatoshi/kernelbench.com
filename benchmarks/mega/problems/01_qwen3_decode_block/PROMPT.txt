I need you to build a MegaQwen-style decode-block megakernel for the RTX PRO 6000 (SM120 Blackwell, GDDR7, 1.8 TB/s). The reference implementation is in reference.py and your kernel goes in solution.py. You can make whatever mess you want in this directory - scratch files, intermediate kernels, build artifacts, profiling traces - but the final answer has to be in solution.py with the same Model, get_inputs, and get_init_inputs interface as reference.py.

The op is one Qwen3-0.6B transformer block decode step for batch=1 and one new token, matching the shape used by https://github.com/Infatoshi/megaqwen. Static dimensions are hidden_size=1024, intermediate_size=3072, num_attention_heads=16, num_kv_heads=8, head_dim=128, q_proj=(2048,1024), k_proj=(1024,1024), v_proj=(1024,1024), o_proj=(1024,2048), gate_proj=(3072,1024), up_proj=(3072,1024), and down_proj=(1024,3072). The forward pass is input RMSNorm; Q/K/V projection; per-head Q/K RMSNorm; RoPE on the current token; append current K/V to the supplied cache; GQA attention decode over seq_len tokens; O projection plus residual; post-attention RMSNorm; gate/up projection; SiLU(gate)*up; down projection plus residual. Inputs are hidden of shape (1024,) bf16, k_cache and v_cache of shape (8, seq_len - 1, 128) bf16, and cos/sin of shape (seq_len, 128) bf16. Return the final hidden vector of shape (1024,) bf16. Correctness tolerance is 0.08 abs/rel on bf16.

The shapes you have to handle are seq_len=32, seq_len=128, and seq_len=512. The projection GEMVs have fixed Qwen3-0.6B dimensions, while the decode attention GEMV/scan size is based on seq_len. This is a decode megakernel problem, so the point is to reduce launch overhead and intermediate traffic across the full block, not to optimize just one GEMM. You do not need to support arbitrary model dimensions, batch sizes, or training.

This needs to be a real custom kernel. Do not call torch.matmul, torch.mm, torch.mv, torch.bmm, torch.einsum, torch.nn.functional.linear, F.linear, scaled_dot_product_attention, or transformers. Try CUDA C++ via load_inline, cooperative groups, Triton, CUTLASS/CuTe pieces, inline PTX, or another real custom-kernel path. If you need ideas, look up the MegaQwen repo, CUDA cooperative groups, PTX docs, CUTLASS sources, or related decode-kernel references yourself.

Your flywheel is implement, profile (ncu, nsys, torch.profiler - whatever is useful) and time it with benchmark.py, verify correctness by running `python check.py` and reading the output, then iterate. Do not substitute your own one-off correctness snippets for check.py - it iterates over every shape and seed, your spot-check almost certainly will not. If `python check.py` has not printed PASS, you are not done. Take as long as you need to actually push the number up.
