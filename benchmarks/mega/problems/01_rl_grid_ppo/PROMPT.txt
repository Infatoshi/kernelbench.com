I need you to make a from-scratch PPO training run go as fast as possible on the RTX PRO 6000 (SM120 Blackwell, GDDR7, 1.8 TB/s). The reference implementation is in reference.py and your version goes in solution.py. You can make whatever mess you want in this directory - scratch files, custom kernels, build artifacts, profiling traces - but the final answer has to be solution.py exposing the same entry point as reference.py:

    train(total_env_steps: int, seed: int) -> list[float]

It runs one full PPO training run from scratch, fully determined by seed, and returns the per-iteration mean episodic return (the learning curve). You are graded on training throughput - environment steps per second over the whole run - so the goal is to keep the entire loop resident on the GPU and cut the per-step launch and dispatch overhead that the naive reference pays. This is the megakernel idea applied to RL: env step, policy forward, action sampling, advantage computation, and the PPO update all want to be fused, not a Python loop firing hundreds of tiny kernels per step.

The task is fixed and defined by reference.py; match it exactly. The environment is vectorized grid foraging: NUM_ENVS=4096 independent agents on an 11x11 board, action 0/1/2/3 = up/down/left/right (clamped to the board), reward +1 for stepping onto the food, which then respawns at a uniform random cell. The whole board re-randomizes every HORIZON=32 steps. The observation is 4 floats: (food_x - agent_x)/11, (food_y - agent_y)/11, agent_x/10, agent_y/10. The policy is a tiny actor-critic MLP: Linear(4,64), tanh, then a Linear(64,4) action head and a Linear(64,1) value head. PPO uses rollout length 32, gamma 0.99, GAE lambda 0.95, clip 0.2, 4 epochs of 4 minibatches, Adam lr 3e-3, entropy coef 0.01, value coef 0.5, grad-norm clip 0.5. Each training run does total_env_steps environment steps total (40 iterations of rollout 32 across 4096 envs). The reference is the source of truth for every one of these details.

Correctness is not bit-exact - it is the learned return level. check.py trains your solution and the reference from scratch on several seeds and requires your final-window mean return to land in a band around the reference's, and to have actually climbed from your own early-iteration baseline. A run that skips or weakens the learning to go fast will fail. So will one that quietly makes the environment easier (it would overshoot the band).

Do not import a ready-made RL stack or environment: no gym, gymnasium, pufferlib, cleanrl, stable_baselines3, tianshou, envpool, sample_factory, rllib, or torchrl. You implement the env and PPO yourself. Write the fused kernels yourself - Triton (@triton.jit) or custom CUDA via load_inline. This is a megakernel benchmark and that is a hard requirement: the training loop must run as genuine fused kernels, not a Python loop firing hundreds of tiny kernels per step, and not a launch-overhead workaround. Do not use torch.cuda.CUDAGraph / graph capture or torch.compile to paper over unfused kernels - a post-run authenticity judge reads your final solution.py (and every local module it imports) and rejects graph/compile/per-step-loop escape hatches, and obfuscating them (getattr string tricks, runtime codegen, sidecar modules) is itself a red flag, so don't; check.py separately hard-fails on importing a prebuilt library. Unlike the token-decode problem you do NOT have to collapse everything to one launch: the thing being ruled out is a per-env-step (or per-minibatch) kernel loop, so fuse coarsely. The whole rollout - all HORIZON env steps across every env, the policy forward, action sampling, reward, and advantage/GAE computation - must run inside a single persistent kernel launch (do many rollout steps and many envs inside one launch), and the PPO update should be fused into that same launch or a small fixed number of extra launches per iteration. What matters is that the kernel-launch count does not scale with env-steps, horizon, or minibatches: a handful of big fused launches per training iteration, not hundreds of small ones. If you want ideas, look up how PufferLib, Brax, or Isaac Gym keep RL loops on-GPU, and Triton persistent-kernel fusion yourself.

Your flywheel is implement, profile (ncu, nsys, torch.profiler - whatever helps) and time it with benchmark.py, verify correctness by running `python check.py` and reading the output, then iterate. Do not substitute your own one-off return-curve checks for check.py - it trains against the reference on every grading seed and your spot-check almost certainly will not. If `python check.py` has not printed PASS, you are not done. Take as long as you need to actually push the throughput up.
