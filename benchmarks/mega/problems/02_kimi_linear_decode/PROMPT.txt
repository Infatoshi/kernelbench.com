I need you to make batch-1 autoregressive decode of a Kimi-Linear hybrid block as fast as possible on the RTX PRO 6000 (SM120 Blackwell, GDDR7, 1.8 TB/s). The model is W4A16: the big projection weights are stored as 4-bit integers, the activations and accumulation are bf16. reference.py is the correctness oracle and baseline.py is an already-optimized pure-PyTorch version; your job is to go beyond baseline.py. Your answer goes in solution.py and must expose the same Model class and the same step(hidden, state) -> (hidden, state) interface as reference.py, with identical buffer/parameter names so it loads the reference weights. Make whatever mess you want in this directory otherwise.

The unit is one repeating motif of Kimi-Linear-48B-A3B: four layers in the pattern [KDA, KDA, KDA, MLA], each followed by a 64-expert MoE FFN (top-8 routed + 1 shared, SiLU, intermediate 1024), with RMSNorm and residuals. Hidden size is 2304. The KDA layers are gated-delta linear attention with 32 heads of dim 128, a short causal depthwise conv (kernel 4) on q/k/v, a per-channel log-decay gate, and a per-head write strength; they carry a fixed-size recurrent state S[32,128,128] plus the conv window across decode steps. The MLA layers are multi-head latent attention with 32 heads, a kv_lora_rank of 512, qk_nope 128, qk_rope 64, v_head 128, decoupled RoPE (theta 10000); they carry a growing compressed latent KV cache (the 512-d latent plus the 64-d rope key per past token).

The weights are quantized W4A16, AWQ/GPTQ-style: per linear, the (in, out) weight is stored as int4 packed two-per-byte along `in` (w_q, uint8, shape (in//2, out)), with per-group asymmetric dequant params group-size 128 along `in` (scales and zeros, bf16, shape (in//128, out)). Dequant is w[k,n] = (unpack(w_q)[k,n] - zeros[k//128,n]) * scales[k//128,n]. The 64 MoE experts (and the shared expert) are stored the same way, one int4 weight set per expert. reference.py has the exact pack/unpack/dequant code, the layer math, the state layout, and how step consumes one token and updates both kinds of state. Read it.

At batch 1 this decode is bandwidth-bound on the weight stream, and the int4 weights are a quarter of the bytes of bf16 - but only if you never materialize the dequantized bf16 weight. reference.py and baseline.py both unpack each weight to a full bf16 matrix and then matmul, which throws the bandwidth advantage away (int4 read plus bf16 write plus bf16 read). The job is to fuse the int4 unpack and per-group dequant directly into the GEMV so you stream the int4 weights once. You are decoding one new token against a context that is already `context_len` tokens long (the states and caches are pre-filled). Correctness is the next-token hidden vector and the updated decode state matching reference.py by cosine similarity (>= 0.98), checked from an identical fed state across several seeds and context lengths. You are scored on decode latency relative to baseline.py, geomean over context lengths 2048, 8192, and 16384.
