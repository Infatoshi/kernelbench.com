I need a sort-free top-p (nucleus) mask kernel in CUDA for the H100 SXM (SM90 Hopper, HBM3). The reference implementation is in reference.py and your kernel goes in solution.py. You can make whatever mess you want in this directory — scratch files, intermediate kernels, build artifacts, profiling traces — but the final answer has to be in solution.py with the same Model, get_inputs, and get_init_inputs interface as reference.py.

The op: given logits of shape (B, V) fp32 and a nucleus mass p, compute probs = softmax(logits) per row, order tokens by probability descending (ties by lower index), and keep the minimal prefix whose cumulative mass reaches p — the crossing token included, top-1 always kept. Output is a bool mask (B, V). The reference does it with torch.sort; you may not. Forbidden in your solution: torch.sort, torch.argsort, torch.msort, torch.topk, torch.kthvalue, torch.quantile (and their tensor-method forms), cub::DeviceRadixSort, thrust::sort, and Triton — this is KernelBench-Mini's CUDA-only track, so write real CUDA C++ (load_inline, a .cu extension, or inline PTX). A hand-written device-side selection scheme is fine; calling a library sort is not. The intended shape of a fast answer is a histogram or threshold-search over probability space — find the cutoff without materializing a sorted order — but that's a suggestion, not a mandate. Anything you're uncertain about, look it up yourself.

Grading is exact, not tolerance-based: check.py builds a float64 oracle and every token whose exclusive cumulative mass is more than 1e-3 away from p is forced — only tokens inside that thin boundary band are free, which absorbs fp32 summation-order rounding and nothing else. The shapes you have to handle are B=1 V=151936 p=0.9 (single-stream decode, Qwen vocab), B=8 V=128256 p=0.95 (small serving batch, Llama-3 vocab), B=64 V=32000 p=0.9 (large batch), and B=16 V=100003 p=0.92 (odd vocab, no power-of-two tricks). At B=1 this is launch-overhead territory: the scoreboard for this problem is milliseconds against the eager sort-based path, and benchmark.py prints that speedup directly.

Your flywheel is implement, profile (ncu, nsys, torch.profiler — whatever's useful) and time it with benchmark.py, verify correctness by running `python check.py` and reading the output. Don't substitute your own one-off correctness snippets for check.py — it covers every shape and stress case, your spot-check almost certainly won't. If `python check.py` hasn't printed PASS, you're not done. Take as long as you need to actually push the number up.
