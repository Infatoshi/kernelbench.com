I need you to write a fast tensor-parallel all-reduce + residual for an 4xH100 SXM node (NVLink4, ~900 GB/s/GPU, fully connected via NVSwitch). This runs across all 4 GPUs under torchrun. The reference is in reference.py and your kernel goes in solution.py. Make whatever mess you want in this directory — scratch kernels, build artifacts, nsys traces — but the final answer must be in solution.py with the same Model, get_inputs, and get_init_inputs interface as reference.py, and it must work when launched on world_size=4.

The op: each rank holds its own (tokens, hidden) bf16 tensor x — its partial slice of a row-parallel matmul output. You must sum x across all 4 ranks (an all-reduce, SUM) and then add this rank's residual, returning bf16. Read reference.py: it defines exactly this. Each rank's inputs are DIFFERENT (the harness seeds every rank distinctly), so you cannot assume the ranks hold the same data — you must actually move and sum the bytes across the NVLink fabric.

The whole point of this benchmark is the inter-GPU communication, not single-GPU compute. You are graded purely on busbw — achieved NVLink bus bandwidth as a fraction of peak — geomean over the shapes. So calling NCCL is pointless and is in fact forbidden: do NOT use torch.distributed.all_reduce, dist.all_reduce, the c10d functional collectives, or any NCCL collective. Implement the cross-GPU sum yourself with fine-grained NVLink primitives. A one-shot all-reduce (each rank reads peers' buffers from symmetric memory and reduces locally) wins at small sizes; a two-shot / reduce-scatter-then-all-gather wins at large sizes. Look into CUDA symmetric memory (torch symmetric memory / multimem), NVSHMEM, or ParallelKittens — investigate the headers and docs yourself.

Correctness tolerance on the bf16 output is 2.5e-2 rel (abs term scaled to the output rms), checked across every shape, 5 rank-distinct trials, and numeric-stress rescales. The shapes you must handle: tokens=4096 hidden=8192 (large, ~64 MB, bandwidth-bound), tokens=512 hidden=8192 (transitional), tokens=64 hidden=8192 (small, ~1 MB, latency-bound — one-shot territory), and tokens=4096 hidden=8191 (off-alignment hidden, forces you to handle a non-power-of-two width).

Your flywheel: implement, profile (nsys/ncu for NVLink traffic and kernel overlap), time it with benchmark.py, verify with `python check.py`, then iterate.
